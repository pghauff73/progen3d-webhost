ProGen3D Firebase Implementation Scheme
=======================================

Last updated: 2026-03-29


1. Overview
-----------

This application uses Firebase as the primary backend for:
- Authentication
- User records
- File records and grammar content
- AI usage and credit accounting
- Builtin rule library storage
- Private user texture storage and metadata

The Firebase implementation is centralized in:
- /var/www/includes/firebase.php

The application uses the Firebase Admin SDK in PHP through:
- kreait/firebase-php

It also includes:
- google/cloud-firestore

The app mixes:
- Firestore document storage
- Firebase Storage object storage
- Firebase Authentication identity verification

3. Authentication Scheme
------------------------

Purpose
- Use Firebase Auth as the identity authority.
- Sync authenticated Firebase users into local user records stored in Firestore.

Main authentication flow
1. Browser authenticates through Firebase client SDK.
2. Browser sends Firebase ID token to the PHP backend.
3. PHP verifies the ID token through firebase_verify_id_token().
4. PHP syncs identity into a local user record.
5. PHP stores Firebase uid in the PHP session.

Main helpers
- firebase_verify_id_token()
- firebase_extract_bearer_token()
- sync_firebase_identity_to_local()
- current_user()

Related endpoints
- /var/www/api/session.php
  - action=login
  - action=logout
  - action=status

Session behavior
- If a bearer token is present, the backend verifies it and syncs the local session.
- If no bearer token is present, the backend falls back to the existing PHP session.

User authorization model
- Role is stored in user records.
- normalize_user_role() standardizes role names.
- is_admin() is used for admin-only features.

Bootstrap admin promotion
- firebase_promote_bootstrap_admin_if_needed()
- Automatically grants admin custom claims to the configured bootstrap admin email once verified.

5. Firestore Transport Strategy
-------------------------------

The app supports two Firestore access modes:
- SDK-based Firestore access
- REST fallback access

Why both exist
- SDK is preferred for richer features, especially transactions.
- REST fallback exists for compatibility and resilience.

Key transport helpers
- firebase_firestore_database()
- firebase_firestore_prefer_rest()
- firebase_firestore_rest_* helpers

Pattern used throughout firebase.php
1. Try SDK collection/document/query path.
2. If unavailable or stack-overflow-like Firestore failure occurs, fall back to REST helpers.

Important caveat
- Some features require the SDK transaction path and are not fully supported with REST-only operation.
- The AI credit system uses Firestore transactions for correctness.

13. OpenAI-Backed Texture Generation Scheme
-------------------------------------------

Purpose
- Generate textures directly from prompts and store them in user slots.

Endpoint
- /var/www/api/textures.php?action=generate

OpenAI usage
- Calls:
  - POST https://api.openai.com/v1/images/generations

Model
- Default:
  - gpt-image-1
- Configurable through:
  - OPENAI_IMAGE_MODEL

Flow
1. User submits prompt for a selected slot.
2. Server sends constrained texture prompt to OpenAI Image API.
3. Image bytes are returned as base64.
4. Server normalizes image to 512x512 PNG.
5. Server stores object in Firebase Storage.
6. Server writes/upserts metadata in Firestore.

16. Design Principles In This Scheme
------------------------------------

1. Keep Firebase Auth as identity authority.
2. Keep app-specific user state in Firestore.
3. Store large/raw content in Firebase Storage, not Firestore.
4. Use Firestore documents for metadata, indexing, and audit records.
5. Keep browser UI display-only for sensitive accounting.
6. Use transactions where correctness matters, especially for credits.
7. Support fallback paths where practical, but rely on SDK transactions for high-integrity flows.
8. Use fixed naming and predictable storage paths where the grammar/runtime must reference assets directly.
